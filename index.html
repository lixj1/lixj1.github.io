Prompt / 项目文件清理

针对历史项目目录的保留式整理提示词

用于整理历史项目目录，只保留运行必需内容与核心数据。

推荐 skill：planning-with-files
适用对象：历史项目整理
输出目标：保留清单 + 删除建议 + 风险说明
你希望模型做到什么

先盘点，再判断依赖，再给出删除方案，不要上来就粗暴清空目录。

保留范围

一键启动后的所有必需程序、data 目录、与 1316-305 项目直接相关的文件。

删除范围

测试程序、临时脚本、历史无关数据、与 1316-305 无关的残留资源。

可直接复制的 Prompt

请帮我整理当前文件夹下的项目代码和数据，只保留目前一键启动后所有程序真正需要的内容，以及 data 目录和与 1316-305 项目直接相关的文件。 请按下面步骤执行： 1. 先完整扫描当前目录结构，列出主要文件夹、脚本、配置文件、数据目录和可能的启动入口。 2. 识别“一键启动”真正依赖的程序、配置、资源文件和数据，不要凭文件名猜，要根据引用关系、导入关系、配置指向和运行路径判断。 3. 单独识别与 1316-305 项目直接相关的代码、文档、数据和配置。 4. 把以下内容归为“建议移除或归档”候选： - 与 1316-305 项目无关的数据 - 测试用程序、临时脚本、废弃脚本 - 重复文件、旧版本备份、无引用资源 - 仅用于实验且当前启动流程完全不依赖的内容 5. 输出结果时必须分成四部分： - 必须保留 - 建议归档 - 建议删除 - 暂时不确定，需人工确认 6. 每一项都要写出判断依据，尤其说明它和启动流程或 1316-305 项目的关系。 7. 最后给出一个“最小可运行目录方案”，说明清理后目录建议如何组织。 注意： - 不要直接删除文件 - 不要在依据不足时武断判断 - 优先做保留式整理，而不是激进删除 - 如果发现某些文件看起来无用，但可能被动态加载、配置引用或外部流程依赖，必须标记为“需人工确认” - 如果合适，请采用 planning-with-files 这种先规划文件级调整、再执行的工作方式

使用提醒

  • 这类任务最怕模型只看文件名，不看真实引用链，所以 Prompt 里已经明确要求按依赖关系判断。
  • 先让模型出清单，再让它根据你的确认逐步移动或删除，风险会小很多。
  • 如果项目里有动态 import、脚本拼路径、配置注入这类机制，一定要保留“需人工确认”这一档。

如何使用 Skill

  • 把 planning-with-files 整个文件夹放到当前项目的 .copilot/skills 目录下。
  • 推荐目录结构：.copilot/skills/planning-with-files/SKILL.md。
  • 打开 VS Code 的 Copilot 对话后，直接说明“请使用 planning-with-files 帮我先做文件级整理规划”。
  • 如果当前环境支持 slash 形式，也可以直接尝试输入 /planning-with-files。
  • 团队分享时不要只发 SKILL.md，最好把整个 planning-with-files 文件夹一起发给大家，避免示例或模板缺失。

推荐输出形态

  • 目录树概览
  • 启动关键链路
  • 1316-305 相关文件清单
  • 可删项与风险等级
  • 最终整理后的目录结构建议
Prompt / 需求梳理与任务拆解

把模糊需求整理成能直接开工的执行清单

用于把模糊需求整理成可执行的任务清单。

适用对象：新需求启动
目标输出：理解摘要 + 开放问题 + 任务清单
适合放到协作群或工单前置阶段
适合什么时候用

当需求来自口头描述、聊天记录、截图、零散文档，且还没有形成统一执行口径时。

模型先做什么

先总结目标与非目标，再列假设、风险和待确认问题，最后才进入任务拆分。

避免什么坑

避免模型直接脑补需求细节，或者在需求未锁定前就开始拍脑袋设计方案。

可直接复制的 Prompt

我现在有一项新需求，但信息比较分散，可能来自聊天记录、口头说明、草稿文档和现有页面。请你不要直接开始实现，而是先帮我完成需求梳理和任务拆解。 请按下面顺序输出： 1. 用尽量准确的语言总结这次需求到底要解决什么问题。 2. 明确目标用户、使用场景、约束条件和明显的非目标。 3. 列出你当前看到的所有假设，不要默认这些假设是真的。 4. 列出必须先确认的开放问题，并标明如果不确认会带来什么风险。 5. 在上述内容基础上，把需求拆成可以执行的任务清单，要求每个任务都尽量独立、可验证、可交付。 6. 对每个任务补充：输入、输出、依赖项、风险点、建议负责人类型。 7. 最后给出一个建议执行顺序，说明为什么这个顺序更合理。 注意： - 不要跳过“理解需求”和“提出问题”这两个阶段 - 不要在信息不足时擅自补完整个方案 - 如果需求里有明显冲突的地方，请单独列出来 - 输出尽量结构化，方便直接转成实施计划或任务单

推荐输出形态

  • 需求摘要
  • 假设列表
  • 开放问题
  • 任务拆解清单
  • 优先级和执行顺序
Prompt / 旧项目接手与代码读盘

快速读懂别人留下来的旧项目，而不是在杂乱目录里盲目排查

用于快速读懂旧项目的启动链路、模块结构和风险点。

适用对象：历史项目接手
目标输出：启动链路 + 模块地图 + 风险热点
适合接手初期快速读盘
先看什么

先找启动入口、配置文件、环境变量、依赖声明和主流程调用链，不要一开始就全仓库漫游。

核心输出

要拿到项目结构图、关键模块职责、核心数据流、已知风险和仍不明确的区域。

最常见误区

只做文件罗列，不做业务链路解释；或者只看静态结构，不看实际运行入口和配置依赖。

可直接复制的 Prompt

我现在要接手这个旧项目，请你帮我先做“代码读盘”和“接手地图”，目标是在最短时间内理解它如何启动、核心逻辑在哪、哪些地方风险最大。 请按下面步骤分析： 1. 先识别项目的启动入口、主程序、配置文件、依赖文件和环境变量来源。 2. 梳理从启动到主功能执行的关键调用链，说明程序是怎么跑起来的。 3. 根据目录结构和引用关系，总结主要模块分别负责什么。 4. 找出和业务强相关的核心模块、数据流向、外部依赖、定时任务、脚本入口或批处理入口。 5. 标记高风险区域，例如：耦合过深、命名混乱、配置分散、重复逻辑、疑似废弃但可能仍在使用的代码。 6. 输出时分成以下几部分： - 项目启动方式 - 核心模块地图 - 关键业务链路 - 风险热点 - 暂不明确、需要人工确认的区域 - 建议的接手阅读顺序 注意： - 不要只列目录，要解释每个关键部分在系统中的作用 - 如果有动态加载、脚本拼路径、数据库配置或外部服务依赖，要明确指出 - 如果发现文档缺失或启动流程不闭合，要直接标成风险项

推荐输出形态

  • 项目启动说明
  • 核心目录职责图
  • 主业务流程梳理
  • 高风险模块与接手建议
  • 优先阅读顺序
Prompt / 报错定位与根因分析

遇到报错时，先找根因，不要只补表层

用于从报错现象收敛到真正根因，而不是只做表层修复。

适用对象：线上或本地异常排查
目标输出：根因假设 + 验证步骤 + 修复建议
适合配合日志和报错栈使用
模型应该先做什么

先区分“现象”“直接触发点”“可能根因”，不要把第一条报错就当最终原因。

要补哪些证据

报错栈、最近改动、输入数据、运行环境、日志片段、复现步骤，缺任何一个都会影响判断质量。

避免什么坑

避免只给“可能是缓存、网络、版本问题”这类空泛答案，而不说明验证路径。

可直接复制的 Prompt

我现在遇到一个报错或异常现象，请你帮我做根因分析，而不是只给表面修复建议。 我会提供：报错信息、日志片段、复现步骤、相关代码、最近改动和运行环境。请你按下面顺序分析： 1. 先区分当前信息里的“现象”“直接报错点”“潜在根因”，不要把它们混为一谈。 2. 根据报错栈、调用链、上下文代码和输入条件，列出最可能的 3 到 5 个根因假设。 3. 对每个根因假设都说明：为什么怀疑它、有哪些证据支持、还缺什么证据。 4. 给出最小验证步骤，帮助我快速排除错误方向。 5. 如果最终能收敛到最可能根因，请给出修复建议，并说明这是“根治”还是“临时止血”。 6. 最后把输出整理成： - 问题现象 - 最可能根因 - 验证步骤 - 修复建议 - 风险与回归检查项 注意： - 不要只根据报错第一行下结论 - 不要输出过于泛化的废话式排查建议 - 如果信息不足，请明确指出还需要哪些信息 - 优先找最可能导致反复复发的问题根因

推荐输出形态

  • 问题现象摘要
  • 根因假设排序
  • 验证步骤清单
  • 最终修复建议
  • 回归测试检查项
Prompt / 接口联调与模拟数据生成

接口没好也能推进联调，把契约和假数据先补齐

用于补齐接口契约、联调样例和可直接使用的 Mock 数据。

适用对象：前后端联调阶段
目标输出：接口契约 + Mock 数据 + 边界案例
适合接口文档不全时补位
模型先补什么

先整理接口目的、字段定义、请求响应结构，再生成可直接用于联调的示例和假数据。

真正有价值的输出

不是一份漂亮的 JSON，而是正常、异常、空状态、边界状态都覆盖到的联调素材。

常见问题

字段命名不统一、状态码语义模糊、分页字段不稳定、列表空态没定义、错误响应没约定。

可直接复制的 Prompt

我现在要做接口联调，但接口文档可能不完整，或者后端还没完全准备好。请你帮我先整理接口契约，并生成能用于前端联调和测试的模拟数据。 请按下面顺序输出： 1. 根据我提供的接口说明、字段草稿、页面需求或已有代码，先总结这个接口要解决什么业务问题。 2. 列出请求方法、路径、请求参数、响应字段、字段含义、类型、是否必填。 3. 如果字段定义存在歧义或缺失，请单独列出待确认问题。 4. 生成以下几类模拟数据： - 正常成功返回 - 空数据返回 - 边界条件返回 - 错误响应返回 5. 如果适合，请补充前端联调时最需要关注的状态切换，比如 loading、empty、error、partial data。 6. 最后输出一份结构化联调清单，方便前后端对齐。 注意： - 不要只给一份理想化成功数据 - 字段命名要尽量统一、可落地 - 如果某些字段明显存在业务歧义，请直接标记出来 - 优先生成能直接喂给页面调试的示例数据

推荐输出形态

  • 接口契约表
  • 待确认字段清单
  • 成功与异常 Mock 数据
  • 页面状态覆盖清单
  • 联调对齐问题列表
Prompt / 前端页面重构

不推翻功能，先把页面结构、样式和可维护性拉回正轨

用于在不破坏现有功能的前提下整理页面结构与可维护性。

推荐 skill：frontend-design
适用对象：老页面重构
目标输出：重构方案 + 拆分建议 + 风险点
先稳什么

先识别现有页面的核心功能和用户操作链路，确保重构过程中不破坏关键行为。

再拆什么

优先拆重复样式、超长组件、混乱状态管理和无层级的 DOM 结构，再考虑视觉升级。

避免什么坑

避免只做视觉翻新却不解决结构问题，也避免为了“重构漂亮”而一次改动过大导致行为回归。

可直接复制的 Prompt

我现在要重构一个已有前端页面。它已经能运行，但结构混乱、样式重复、可维护性差。请你不要直接推翻重写，而是先给我一个稳妥的重构分析和执行建议。 请按下面顺序处理： 1. 先总结这个页面当前承担的核心功能、主要交互路径和关键状态。 2. 找出最影响维护性的结构问题，例如：组件过长、样式散落、状态耦合、命名混乱、重复布局、语义结构不清。 3. 把问题拆成几类： - 必须先修的结构问题 - 可以顺手整理的样式问题 - 可以后续再做的体验升级项 4. 给出一个分阶段重构方案，每阶段都说明目标、风险和验收方式。 5. 如果适合，请给出推荐的组件拆分方式、样式组织方式和命名建议。 6. 输出时尽量结构化，最后给出“最低风险改造顺序”。 注意： - 不要一上来就建议全部重写 - 优先保证现有功能不回归 - 如果页面已有明确设计语言，重构时应优先保持一致 - 如果某些问题只是表象，请继续追到结构根因

推荐输出形态

  • 页面职责摘要
  • 结构问题清单
  • 分阶段重构计划
  • 组件拆分建议
  • 最低风险执行顺序
