推荐使用时机

内容边界

当前页面只聚焦 Git 存档，不扩展到 PR、rebase、冲突处理等更广泛的 Git 协作内容。

关键命令摘要

如果你看完视频后想快速回忆核心动作，可以直接先看这组命令。它们对应的是“查看状态 → 暂存 → 提交 → 推送”这一条最常用的 Git 存档路径。

git status

先看当前工作区和暂存区状态，确认有哪些文件发生了变化。

git add .

把当前需要留档的改动加入暂存区。如果只想存部分文件，也可以改成逐个文件 add。

git commit -m "阶段性存档：补充 Git 方法页面"

生成一次明确可追踪的提交记录。提交说明建议写清当前阶段做了什么，后续回看会更容易。

git push origin main

把当前提交推到远端仓库，完成真正意义上的留档。如果你的分支不是 main，请替换成对应分支名。

Prompt / 项目文件清理

项目目录保留式清理

分享者：李晓娟

只保留运行必需内容与核心数据。

推荐 skill：planning-with-files
适用对象：历史项目整理
输出目标：保留清单 + 删除建议 + 风险说明
你希望模型做到什么

先盘点，再判断依赖，再给出删除方案，不要上来就粗暴清空目录。

保留范围

一键启动后的所有必需程序、data 目录、与 1316-305 项目直接相关的文件。

删除范围

测试程序、临时脚本、历史无关数据、与 1316-305 无关的残留资源。

可直接复制的 Prompt

请帮我整理当前项目目录，只保留一键启动真正依赖的程序、data 目录，以及与 1316-305 项目直接相关的文件。 请先完整扫描目录，再完成以下工作： 1. 识别启动入口、核心脚本、配置文件、资源文件和数据目录，并根据引用关系、导入关系、配置指向和运行路径判断真实依赖。 2. 单独梳理与 1316-305 项目直接相关的代码、文档、数据和配置。 3. 把无关数据、测试程序、临时脚本、废弃脚本、重复文件、旧备份、无引用资源等内容列为“建议归档或删除”候选。 4. 输出时分成：必须保留、建议归档、建议删除、暂不确定需人工确认。 5. 每一项都要写清判断依据，尤其说明它与启动流程或 1316-305 项目的关系。 6. 最后给出一个“最小可运行目录方案”。 约束： - 不要直接删除文件 - 不要仅凭文件名判断 - 优先保留式整理，而不是激进删除 - 如存在动态加载、配置引用、脚本拼路径或外部流程依赖，必须标记为“需人工确认” - 如果合适，请采用 planning-with-files 这种先规划文件级调整、再执行的方式 输出时请给出： - 目录树概览 - 启动关键链路 - 1316-305 相关文件清单 - 可删项与风险等级 - 整理后的目录结构建议

使用提醒

  • 这类任务最怕模型只看文件名，不看真实引用链，所以 Prompt 里已经明确要求按依赖关系判断。
  • 先让模型出清单，再让它根据你的确认逐步移动或删除，风险会小很多。
  • 如果项目里有动态 import、脚本拼路径、配置注入这类机制，一定要保留“需人工确认”这一档。

  • 如果当前环境支持 slash 形式，也可以直接尝试输入 /planning-with-files。
  • 可直接查看完整文件夹：planning-with-files

方法一 / 角色持久化

用 copilot-instructions.md 给 AI 一个永久角色

分享者：徐艺嘉 ｜ 更新时间：2026-04-17 ｜ 适用场景：跨项目、跨 Session 保持一致的 AI 协作风格与专业背景 ｜ 关键词：copilot-instructions.md、角色设定、全局配置、用户级指令

不用每次对话都重新介绍自己的背景——把角色写进配置文件，Copilot 在每次开口前就已经知道它在和谁协作、应该用什么姿态回应。

问题是什么

每次新开一个 Session，AI 对你的背景、领域和工作方式一无所知。你要反复解释"我是药企的医学经理"、"我不是专业程序员"，对话才能找到节奏，效率很低。

关键思路

VS Code 会在每次对话前自动加载 copilot-instructions.md 文件的内容并注入上下文，相当于给 Copilot 一份"入职手册"——无需你开口，它已经知道该扮演什么角色。

两种生效范围

放在用户配置目录（%AppData%/.../Code/User/prompts/）则全局生效，所有项目通用；放在某个项目的根目录则只对该项目生效，互不干扰。

我们设定的角色是什么

这份配置文件将 Copilot 的角色设定为一位同时具备多重视角的复合型合作者，能在一次对话里无缝切换技术实现与业务需求两个维度：

资深全栈工程师

负责代码实现、架构思考和工程规范。遇到技术问题时，以工程师视角给出可落地的方案，而不是泛泛而谈。

产品经理 + 医学经理

理解需求背后的业务逻辑和药企场景边界。药企的数据合规、功能优先级、汇报逻辑，都在这个角色的认知范围内。

药企统计师

熟悉临床数据结构、统计分析规范和医学报告惯例。处理数据相关任务时能给出符合行业标准的建议，而不是通用的数据科学思路。

如何配置

1. 找到配置目录

按 Win + R，输入 %AppData% 并回车，依次进入 Code → User → prompts 文件夹。如果 prompts 文件夹不存在，手动新建即可。

2. 新建文件

在 prompts 文件夹内新建一个文本文件，命名为 copilot-instructions.md，用记事本或 VS Code 打开后，把角色设定内容粘贴进去并保存。

3. 重启 VS Code 生效

保存文件后重新启动 VS Code，之后每次打开 Copilot Chat，这份角色设定就会自动注入——无需任何额外操作，也无需在对话里手动引用它。

配置文件内容示例

将以下内容复制到你的 copilot-instructions.md 文件中，根据实际情况调整细节：

copilot-instructions.md

你是一位资深全栈工程师兼产品经理，同时具备药企医学经理和统计师的专业背景。 你的协作原则： - 技术问题优先给出可落地的实现方案，避免理论堆砌 - 涉及业务需求时，主动从药企合规、医学逻辑、统计规范角度审视 - 用简洁、直接的语言回应，不要过度解释显而易见的内容 - 遇到模糊需求时，先确认核心目标再动手，不要乱猜 - 代码产出需附带最小必要注释，方便非专业背景同事阅读

为什么有效

角色持久化解决的是 AI 协作中的"每次归零"问题。一个没有背景的 AI 会给出泛用答案；一个持续扮演同一角色的 AI 会记住你的工作语境，输出更贴近实际需要的内容。对团队来说，这份配置还有额外价值：它是一份可以共享的"协作契约"——把它放进项目根目录，所有成员的 Copilot 就会用同一套标准与项目交互，减少因 AI 输出风格不一致而造成的沟通摩擦。

方法二 / 长 Session 管理

AI 协作转场法

分享者：殷悦 ｜ 更新时间：2026-04-17 ｜ 适用场景：长 Session 协作、复杂项目接力、上下文开始漂移时 ｜ 关键词：handover、context.md、agents-md、上下文管理

核心目标不是让 AI 记住所有对话，而是让它始终站在最新、最干净的项目事实上继续工作。

问题是什么

长 Session 里信息不断堆积，AI 会被旧结论、无效尝试和冗余上下文拖偏，出现上下文漂移，最终表现为逻辑混乱和执行质量下降。

关键思路

不要把“记忆”继续压在对话流里，而是主动把当前有效状态收敛成结构化文档，让 AI 在新 Session 里从唯一事实来源重新接管工作。

适用场景

适合代码改造跨度大、协作周期长、Prompt 多轮迭代后开始出现答非所问或重复返工的项目型任务。

具体怎么做

1. 配置指令

在项目根目录创建 .cursorrules 或 .github/copilot-instructions.md，写入“遵循 getsentry/agents-md 规范，输入 !handover 时更新 context.md”，先把转场动作变成项目约定。

2. 状态固化

当 Session 开始变长、AI 出现疲劳感时，输入 !handover，让它把当前已验证成功的逻辑、残留 Bug 和下一步任务重写进 context.md，而不是继续拖着旧对话硬聊。

3. 无损转场

开启新 Session 后，直接通过 @context.md 引导 AI。这样它会绕开旧对话噪声，基于精简后的项目现状继续推进，避免把历史废话再次带回上下文。

为什么有效

这套方法的本质，是把 AI 的记忆负荷从“对话流”转移到“结构化文档”。对话适合推进，文档适合沉淀；当两者分工明确后，AI 不再需要从冗长上下文里反复打捞有效信息，智力状态会明显更稳定，项目进度也会因此变得更透明、更可回溯。对团队来说，context.md 不只是 handover 文件，更像每一轮协作后的“最高意志”，新 Session 直接从这里接棒即可。

方法三 / 工程级代码质量

Superpowers 插件：让 AI 写出工程级代码

分享者：李晓娟 ｜ 更新时间：2026-04-17 ｜ 适用场景：需要测试覆盖、文档规范、遵循工程最佳实践的项目开发 ｜ 关键词：Superpowers、agent skills、VS Code 插件、工程规范、代码质量

普通 AI 助手能写出"能跑的代码"，Superpowers 的目标是强制 AI 遵循完整的软件开发流程，输出真正可用于生产的工程级代码。

问题是什么

AI 生成的代码往往能运行，但缺少测试、没有文档、不遵循团队规范——像"聪明的实习生"，输出停留在玩具级，无法直接用于生产环境。

关键思路

Superpowers 是由 Jesse Vincent（obra）开发的开源代理技能框架，相当于给 AI 配了一位"资深工程师导师"，通过内置 skills 约束 AI 始终遵循测试、文档、代码审查等完整开发流程。

适用场景

适合需要测试覆盖的功能开发、对外交付的模块文档输出、长期维护型项目，以及对代码质量有明确要求的团队协作。

如何安装

打开 VS Code，点击左侧插件市场图标（Extensions），在搜索框中输入 Superpowers，找到 Jesse Vincent（obra）发布的版本后点击 Install 安装即可。安装后无需额外配置，插件会自动注入到 Copilot Chat 的代理能力中。

在 VS Code 扩展市场搜索 "Superpowers" 即可找到对应插件，点击 Install 后立即生效

具体怎么做

1. 安装并验证

在 VS Code 扩展市场安装 Superpowers 后，打开 Copilot Chat 面板，试着描述一个开发任务，确认 AI 回复中出现测试建议或文档提示，说明插件已生效。

2. 正常使用 Copilot

无需改变提问习惯，照常使用 Copilot Chat 描述功能需求。Superpowers 在后台为 AI 注入工程规范约束，会主动提示补充单元测试、生成模块文档或指出潜在的代码质量问题。

3. 三维验收输出

收到 AI 输出后，对照三个维度检查：✓ 有没有测试覆盖、✓ 有没有必要注释、✓ 是否遵循项目规范。Superpowers 的目标正是把这三项标准内化为 AI 的默认行为，而不是等你事后追单。

为什么有效

大多数 AI 编程助手被设计为"满足用户当前请求"，而不是"遵循软件工程最佳实践"。Superpowers 通过在 agent 层植入可配置的 skill 集合，系统性地改变了 AI 的默认行为边界——它不再只关心"能不能跑"，而是同时关心"测试有没有写、文档有没有同步、接口有没有规范"。对团队来说，这意味着 AI 协作产出可以从"需要人工补课"变成"可以直接进入 Code Review"，大幅降低因 AI 输出质量不稳定带来的返工成本。
