主键列名兼容改造

统一兼容“受试者代码”“参与者代码”等主键列名变体。

适用对象：Python / Streamlit / Pandas 项目
目标输出：统一列名入口 + 下游兼容改造
要求兼容旧数据与新数据
核心目标

先理解模块职责和数据流，再把 subject ID 列统一到一个固定内部列名，避免下游继续硬编码旧列名。

重点范围

优先处理 shared/data_loader.py、app.py 的缓存读取入口，以及独立读 Excel 的 ae_coding.py、gantt_builder.py 等模块。

边界约束

不要改旧存档文件和离线包目录，不改 Excel/Parquet 本体，只改代码中的列名标准化与引用方式。

可直接复制的 Prompt

这是一个 Python + Streamlit + Pandas 项目。当前各模块大量硬编码“受试者代码”作为 DataFrame 主键列名，但新数据使用“参与者代码”，导致下游流程断裂。 请先通读项目，理清模块职责、数据加载入口和主数据流，然后完成以下改造： 1. 全局兼容这些列名变体：受试者代码、参与者代码、受试者编号、参与者编号、Subject ID、Participant ID，以及你发现的其他同类变体。 2. 推荐在 shared/data_loader.py 的 load_study_data() 附近新增 normalize_subject_col()，把所有 DataFrame 统一 rename 成同一个内部列名，下游只使用这个统一名。 3. app.py 中 _cached_read_excel_sheet() 的标准化逻辑也要对齐到同一个内部列名。 4. 对独立读 Excel 的模块，如 shared/ae_coding.py、gantt_builder.py，也要在读取后执行同样的标准化。 5. 重点修改列名引用逻辑，如 df['受试者代码']、df[df['受试者代码'] == id]、'受试者代码' in df.columns；UI 文案里的“受试者”可不改。 约束： - 不要改动 app_v1_checkpoint.py、app_v2_mobile_checkpoint.py、old_mob_ui.py、old3.py、old22.py、old_new.py、old2.py 和 _offline_pkg/ - 必须同时兼容旧数据和新数据 - 不要改 Excel / Parquet 数据文件本身 验证： - 分别用旧数据和新数据验证个体审阅、群体审阅、中心比较、Query 引擎、甘特图、移动端是否正常 输出时请给出： - 主数据加载入口、独立读表模块和受影响范围 - 统一列名方案 - 改动文件、改动函数、修改原因 - 验证步骤、风险点和需人工确认项

Prompt / 项目文件清理

项目目录保留式清理

只保留运行必需内容与核心数据。

推荐 skill：planning-with-files
适用对象：历史项目整理
输出目标：保留清单 + 删除建议 + 风险说明
你希望模型做到什么

先盘点，再判断依赖，再给出删除方案，不要上来就粗暴清空目录。

保留范围

一键启动后的所有必需程序、data 目录、与 1316-305 项目直接相关的文件。

删除范围

测试程序、临时脚本、历史无关数据、与 1316-305 无关的残留资源。

可直接复制的 Prompt

请帮我整理当前项目目录，只保留一键启动真正依赖的程序、data 目录，以及与 1316-305 项目直接相关的文件。 请先完整扫描目录，再完成以下工作： 1. 识别启动入口、核心脚本、配置文件、资源文件和数据目录，并根据引用关系、导入关系、配置指向和运行路径判断真实依赖。 2. 单独梳理与 1316-305 项目直接相关的代码、文档、数据和配置。 3. 把无关数据、测试程序、临时脚本、废弃脚本、重复文件、旧备份、无引用资源等内容列为“建议归档或删除”候选。 4. 输出时分成：必须保留、建议归档、建议删除、暂不确定需人工确认。 5. 每一项都要写清判断依据，尤其说明它与启动流程或 1316-305 项目的关系。 6. 最后给出一个“最小可运行目录方案”。 约束： - 不要直接删除文件 - 不要仅凭文件名判断 - 优先保留式整理，而不是激进删除 - 如存在动态加载、配置引用、脚本拼路径或外部流程依赖，必须标记为“需人工确认” - 如果合适，请采用 planning-with-files 这种先规划文件级调整、再执行的方式 输出时请给出： - 目录树概览 - 启动关键链路 - 1316-305 相关文件清单 - 可删项与风险等级 - 整理后的目录结构建议

使用提醒

  • 这类任务最怕模型只看文件名，不看真实引用链，所以 Prompt 里已经明确要求按依赖关系判断。
  • 先让模型出清单，再让它根据你的确认逐步移动或删除，风险会小很多。
  • 如果项目里有动态 import、脚本拼路径、配置注入这类机制，一定要保留“需人工确认”这一档。

如何使用 Skill

  • 把 planning-with-files 整个文件夹放到当前项目的 .copilot/skills 目录下。
  • 推荐目录结构：.copilot/skills/planning-with-files/SKILL.md。
  • 打开 VS Code 的 Copilot 对话后，直接说明“请使用 planning-with-files 帮我先做文件级整理规划”。
  • 如果当前环境支持 slash 形式，也可以直接尝试输入 /planning-with-files。
  • 可直接查看完整文件夹：planning-with-files
